# 教育政策应用评估与改进建议

生成时间：2026-04-27

## 1. 评估范围

本次评估覆盖以下维度：

- 项目结构与代码边界
- 核心功能与可靠性
- 前端性能与可观测性
- 用户体验、响应式与可访问性

本次评估基于代码审查、构建验证和多 agent 并行分析完成。

已完成的本地验证：

- `npm run build` 通过，产物成功生成到 `dist/`
- `npm.cmd run lint` 未能形成有效检查，因为仓库缺少 ESLint 配置文件

本次评估限制：

- 当前环境没有稳定可用的浏览器自动化链路，无法提供可信的运行时截图证据
- 无法完成真实用户交互录屏、Core Web Vitals 实测和移动端真机验证
- UX 结论主要基于组件结构、样式实现和交互代码

## 2. 执行摘要

这是一个基于 Vite + React 的单页教育时政卡片应用，功能覆盖搜索、收藏、自定义卡片抓取、笔记、题目练习、正文标注和 AI 问答。功能面不小，但当前实现把过多职责堆在 `src/App.jsx`、`src/components/HomePage.jsx`、`src/components/Timeline.jsx` 三个文件中，已经出现结构耦合、交互分裂和性能上限问题。

当前最关键的结论有 8 条：

1. 详情阅读存在两套模型：覆盖层和独立路由并存，但主路径只走覆盖层，导致刷新、分享、返回行为不稳定。
2. 独立详情页本身映射错字段，正文和来源链接对默认官方卡片并不可靠。
3. `HomePage.jsx` 体量过大，承担列表、全文、题目、笔记、AI、标注、拖拽排序等多种职责，是当前维护复杂度和渲染成本的核心来源。
4. 默认新闻全文数据直接内联在主包里，且没有动态拆包，首屏负担偏重。
5. `localStorage` 持久化是同步主线程串行写入，且重复保存大对象，容易在答题、笔记和标注时制造卡顿。
6. AI 问答上下文并不等于“当前卡片上下文”，当前实现只发送前 12 条新闻摘要，导致回答容易泛化。
7. 弹层、浮层和固定工具条大量依赖硬编码位置与全局事件，移动端遮挡、焦点管理和可访问性风险较高。
8. 样式体系处于混用状态：内联样式、Tailwind 风格类名、自写 utility、未接通的 `@apply` CSS 并存，样式管线不稳定。

## 3. 项目结构评估

### 3.1 当前结构特征

- 入口为 `src/main.jsx`
- 应用壳、全局状态、默认数据、结构化逻辑、持久化逻辑集中在 `src/App.jsx`
- 首页列表和主要交互集中在 `src/components/HomePage.jsx`
- 左侧/顶部悬浮导航、自定义卡片新增和另一套 AI 入口集中在 `src/components/Timeline.jsx`
- 文本抓取、判重、题目解析、AI 调用分别在 `src/lib/*`

### 3.2 结构层面主要问题

#### A. 应用级职责过于集中

`src/App.jsx` 既保存默认数据，又负责数据结构化、去重、持久化和路由装配。默认卡片数据从 `src/App.jsx:261` 开始直接内联在组件文件中，`App` 本体从 `src/App.jsx:756` 开始继续承担状态中心角色。

影响：

- 应用入口文件过重
- 默认数据和运行逻辑无法独立演进
- 任何状态或数据形态调整都容易波及主壳

#### B. 首页组件职责失控

`src/components/HomePage.jsx` 约 4000 行，维护 20 多个 `useState`，并同时负责：

- 卡片列表
- 覆盖层详情
- AI 助手
- 题目录入、编辑、排序、做题
- 正文高亮标注
- 富文本笔记
- 来源菜单
- 自定义卡片编辑

影响：

- 本地状态更新面过大
- 理解成本高
- 局部交互易引发整页重渲染
- 回归测试难度高

#### C. 组件之间靠全局事件通信

`Timeline.jsx` 和 `HomePage.jsx` 通过 `openNews`、`newsOverlayChange` 这类自定义事件协作，而不是只通过路由或明确的 React 状态流协作。

相关位置：

- `src/components/HomePage.jsx:863`
- `src/components/HomePage.jsx:906`
- `src/components/Timeline.jsx:357`
- `src/components/Timeline.jsx:780`
- `src/components/Timeline.jsx:880`

影响：

- 页面流转不可预测
- 组件边界不清晰
- 调试和埋点难度增加

#### D. 存在死代码和重复实现

已发现以下重复或未接入内容：

- `src/components/AIAssistant.jsx` 未接入主导入链
- `src/components/NewsList.jsx` 未接入当前主路径
- `getSourceDisplay` 在多个组件中重复实现
- AI 问答在 `HomePage.jsx` 和 `Timeline.jsx` 各实现一套
- 自定义卡片导入/判重在新增和编辑流程中分别实现
- `SearchBar.css`、`NewsList.css` 存在，但组件未导入这些 CSS 文件

### 3.3 结构改进建议

P0：

- 统一详情阅读模型，只保留一套详情容器和一套状态来源
- 把默认数据、状态仓储、业务工具和页面展示从 `App.jsx` / `HomePage.jsx` 中拆开

P1：

- 以功能域拆分 `HomePage.jsx`
- 收敛组件通信方式，逐步替换全局自定义事件

P2：

- 清理未使用组件和未接入样式文件
- 去重展示工具函数和重复的业务流程

## 4. 功能与可靠性评估

### 4.1 高优先级问题

#### 1. 路由详情流没有真正接入主用户路径

证据：

- `src/App.jsx:1084` 定义 `/news/:id`
- 列表点击实际走 `openFullScreen(news)`，见 `src/components/HomePage.jsx:858` 和 `src/components/HomePage.jsx:1832`

表现：

- 从首页点击卡片，地址栏仍停留在 `#/`
- 当前详情页无法被稳定刷新或分享
- 路由页对主路径用户几乎不可达

建议：

- 详情页改成真正的路由页，或改成 route-driven modal，但只能保留一种

#### 2. 独立详情页读取了错误字段

证据：

- `src/components/NewsDetail.jsx:147`、`src/components/NewsDetail.jsx:157` 使用 `news.content`
- `src/components/NewsDetail.jsx:209` 使用 `news.sourceUrl`
- 默认官方卡片主要提供 `verifiedSourceUrl` 和 `references`，见 `src/App.jsx:273`

表现：

- 详情页正文只显示摘要级内容
- 官方卡片来源链接可能为空或无效

建议：

- 详情页正文应以 `detailContent` 为主
- 来源按钮应优先使用 `verifiedSourceUrl` 或第一条有效 `references`

#### 3. AI 助手无法稳定回答“当前这条政策”

证据：

- `HomePage.jsx` 和 `Timeline.jsx` 都把整份 `newsList` 传给 AI
- `src/lib/doubao.js:5` 只截前 12 条新闻
- `src/lib/doubao.js:9` 每条只发送 `news.content.slice(0, 160)`

表现：

- 后排卡片可能根本没进入提示词
- 进入提示词的卡片也只带摘要，没有正文
- 用户在详情页追问具体内容时，回答容易泛化

建议：

- AI 上下文应以当前卡片为第一上下文，必要时再拼接相关卡片
- 默认传入 `detailContent` 的结构化摘要，而不是全局前 12 条摘要

### 4.2 中优先级问题

#### 4. 移动端暴露了不可完成的题目排序能力

证据：

- 入口在 `src/components/HomePage.jsx:2750`
- 排序实现依赖 `draggable`、`onDragStart`、`onDrop`，见 `src/components/HomePage.jsx:4031`

表现：

- 手机端能看到“自定义排序”
- 但大多数移动浏览器不支持这套桌面拖拽路径

建议：

- 为移动端提供显式上移/下移排序，或直接隐藏该入口

#### 5. 搜索不覆盖来源和分类

证据：

- `src/components/HomePage.jsx:1382` 只搜索 `title/summary/content/detailContent`

表现：

- 搜 `考试招生`、`教育技术`、`中国教育考试网` 这类自然关键词时可能得不到结果

建议：

- 把 `source`、`category` 纳入搜索域
- 后续补充大小写、空白和同义词归一化

#### 6. 做题统计已持久化，但刷新后几乎不可见

证据：

- `App.jsx` 会恢复 `questionStats`，见 `src/App.jsx:787`、`src/App.jsx:864`
- 展示却被 `isSubmitted` 限制，见 `src/components/HomePage.jsx:3346`
- `currentArticleStats` 有计算但没有形成持续可见展示，见 `src/components/HomePage.jsx:1413`

表现：

- 统计数据在 localStorage 中存在
- 用户刷新或重开后看不到结果，直到再答一次

建议：

- 将“累计正确率/作答次数”与“本次作答结果”分开显示

## 5. 性能评估

### 5.1 当前基线

本地构建结果：

- `dist/assets/index-He6F45DR.js` 约 `295.05 kB`
- gzip 后约 `98.00 kB`
- `dist/assets/index-IlkOIuZ5.css` 约 `6.91 kB`

项目当前没有发现动态 `import()`，说明基本是单入口单包首屏。

### 5.2 核心性能问题

#### 1. 首包承载过多非首屏能力

证据：

- 默认新闻正文全文内联在 `src/App.jsx:261` 之后
- `App.jsx`、`HomePage.jsx`、`Timeline.jsx` 同步引入了 AI、抓文、题库和复杂详情能力

影响：

- 下载、解析、编译、模块求值全部压在首屏
- 移动端和低性能设备受影响更明显

建议：

- 默认新闻数据单独下沉为静态数据模块或远程资源
- AI、抓文、题库编辑、排序管理等能力按需加载

#### 2. `localStorage` 同步写入过粗

证据：

- `src/App.jsx:877` 到 `src/App.jsx:901` 在多个状态任一变化时连续执行 7 次 `setItem`
- `savedNews` 保存的是完整卡片对象，不是 ID

影响：

- `JSON.stringify` 和 `setItem` 都阻塞主线程
- 笔记、标注、答题等高频交互容易产生卡顿

建议：

- 只保存最小必要数据
- 合并写入策略
- 引入节流/空闲时持久化策略

#### 3. 大组件导致局部更新牵动整页重算

证据：

- `HomePage.jsx` 同时挂载列表、覆盖层、题目、标注、AI、菜单等状态
- `HomePage.jsx:739` 到 `HomePage.jsx:782` 有大量局部状态

影响：

- 收藏、答题、打开详情等操作更容易触发整树重跑

建议：

- 拆分渲染边界
- 把题目、笔记、标注、AI 等独立子系统从主列表树中隔离

#### 4. 卡片列表存在全量重算

证据：

- `HomePage.jsx:1816` 附近每张卡片 render 都会重新计算预览摘要、来源、题目数
- `buildPreviewHighlights` 内部还会做句子扫描和拆分

影响：

- 数据量增长后，收藏和答题也会拖慢列表更新

建议：

- 卡片衍生字段前置计算
- 预览数据缓存化

#### 5. 正文标注算法复杂度会随长度与标注数上升

证据：

- 段落渲染前会按段过滤 marks，见 `HomePage.jsx:2366` 到 `HomePage.jsx:2381`
- 文本偏移计算使用 `TreeWalker`
- 标注应用和合并逻辑会重复扫描 marks

影响：

- 长文和多标注场景会直接拉高交互延迟

建议：

- 预计算段落索引
- 限制单次重排范围
- 建立更稳定的标注数据结构

### 5.3 建议增加的观测点

- 首屏 `LCP`、`INP`、`TBT`
- 首次 React commit 时长
- 单次持久化总耗时
- 单次答题、单次标注的 commit 时长
- `r.jina.ai` 抓文请求的 TTFB、失败率、返回体积
- Doubao 请求体大小、p95 延迟和会话增长曲线

## 6. 用户体验与可访问性评估

### 6.1 主要问题

#### 1. 导航模型分裂

列表通过本地覆盖层打开详情，路由又保留独立详情页，导致：

- 地址与内容状态不同步
- 返回逻辑分裂
- 分享与刷新不可预期

对应位置：

- `src/components/HomePage.jsx:858`
- `src/components/HomePage.jsx:1832`
- `src/App.jsx:1084`
- `src/components/NewsDetail.jsx:75`

#### 2. 关键交互不是稳定的语义控件

已发现以下典型情况：

- 新闻卡片点击以容器点击为主，而不是统一的语义链接/按钮
- 收藏和自定义列表项使用 `div onClick`
- 排序只有拖拽路径，没有键盘替代

影响：

- 键盘可达性弱
- 屏幕阅读器语义不清晰

#### 3. 弹层没有形成完整 dialog 契约

AI、收藏、自定义、添加卡片、全屏详情、排序弹层主要是 `fixed` 容器和遮罩，关闭依赖外点监听，未看到完整的：

- `role="dialog"`
- `aria-modal`
- `Escape` 关闭
- focus trap
- 关闭后焦点归还

#### 4. 移动端布局高度依赖 magic number

当前实现中出现多组固定值：

- 顶部导航 `72px`
- 主区域顶部内边距 `88px`
- 多组底部固定工具条和浮层偏移

风险：

- 小屏设备容易遮挡
- 输入法弹起时更易冲突
- 叠层和滚动锁定更难维护

#### 5. 样式体系不稳定

证据：

- `SearchBar.css`、`NewsList.css` 使用 `@apply`
- `postcss.config.js` 只启用 `autoprefixer`
- 构建后的 CSS 里仍能看到未展开的 `@apply`
- 实际组件大量依赖内联样式和手写 utility

这意味着：

- 响应式规则并不一定真正生效
- 组件样式的一致性不可预测
- 维护者很难判断哪条样式链路才是真正生效的

### 6.2 UX 改进建议

P0：

- 统一详情入口和返回模型
- 抽象统一的弹层原语，补齐语义和焦点管理

P1：

- 重做移动端顶部区和全屏工具条布局
- 搜索区支持窄屏换行或纵向布局
- 为图标按钮、反馈信息、表单错误补齐可访问名称和状态语义

P2：

- 收敛为一套真实可维护的样式系统
- 清理未接通的 `@apply` 和未导入组件 CSS

## 7. 工程与工具链问题

### 7.1 Lint 形同未接入

执行 `npm.cmd run lint` 后，ESLint 报告仓库内找不到配置文件。这意味着当前项目虽然声明了 lint 脚本，但没有真正形成静态检查质量门。

影响：

- 未使用导入、死代码、可疑依赖、可访问性问题难以及时暴露

建议：

- 补齐 ESLint 配置
- 让 lint 成为可执行、可失败的 CI 门禁

### 7.2 样式工具链未闭环

证据：

- 存在 `tailwind.config.js`
- `App.css`、`SearchBar.css`、`NewsList.css` 使用 `@apply`
- `postcss.config.js` 仅启用 `autoprefixer`

建议：

- 明确到底采用 Tailwind 4、手写 CSS，还是混合方案
- 如果保留 Tailwind，就把真正的构建链接通
- 如果放弃 Tailwind，就删除无效 `@apply` 和伪 utility 层

## 8. 优先级路线图

### P0：先解决根因

1. 统一详情页架构，只保留一套详情阅读模型
2. 修正 `NewsDetail` 的正文和来源映射
3. 把 AI 上下文从“全局前 12 条摘要”改为“当前卡片优先”
4. 为弹层建立统一的可访问交互规范
5. 停止在每次状态变化时串行写入多份大对象到 `localStorage`

### P1：降低复杂度和交互成本

1. 拆分 `HomePage.jsx`
2. 把默认新闻数据从主组件中移出
3. 为题目排序提供移动端和键盘可用方案
4. 扩充搜索域到来源和分类
5. 让做题累计统计形成持久可见展示

### P2：补齐工程质量与样式系统

1. 接入真实可用的 ESLint
2. 清理未使用组件与重复实现
3. 收敛样式体系和工具链
4. 为性能与交互建立基础打点

## 9. 建议验收标准

建议把后续改造的验收标准明确成可验证结果：

- 打开任意卡片后，地址栏必须对应唯一可分享 URL
- 刷新详情页后，正文、来源和当前上下文不丢失
- AI 追问默认围绕当前卡片正文作答，而不是全局泛答
- 移动端可以完成题目排序，或不再暴露不可完成入口
- 答题、标注、笔记编辑时不出现明显主线程卡顿
- `npm run lint` 必须真正执行并可失败
- 样式构建链中不再残留无效 `@apply`

## 10. 总结

这个项目不是“功能不够”，而是“功能已经超过当前结构所能稳定承载的范围”。现阶段最需要的不是继续叠功能，而是先把导航模型、数据边界、持久化策略和样式/弹层基础设施收紧。只要把这几个根因处理掉，后续的题库、AI、抓文和详情体验都会更容易演进，也更容易测试。
